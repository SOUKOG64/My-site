# Руководство для программистов: FrameFlow

## 1. Что это за проект

FrameFlow - учебный многостраничный сайт для фотографа-любителя. Проект сочетает:

- статический frontend на HTML, CSS и встроенном JavaScript;
- простой Node.js сервер без фреймворков;
- файловое хранение данных в `data/user-data.json`.

Это не SPA и не production-система. Здесь нет базы данных, полноценной аутентификации, ролей и серверной регистрации пользователей.

## 2. Как устроен проект

Корень проекта содержит основные страницы и сервер:

- `index.html` - главная страница, описание проекта, FAQ и мини-квиз.
- `profile.html` - профиль пользователя и раздел "Мои сервисы".
- `gallery.html` - стандартная галерея и загрузка пользовательских фото.
- `login.html` - демонстрационная форма входа.
- `registration.html` - демонстрационная форма регистрации без backend-интеграции.
- `Cofedensi.html` - политика конфиденциальности.
- `Contats.HTML` - страница контактов.
- `Mod_gallery.html` - альтернативная галерея-слайдер.
- `Mod_gallery_dark.html` - темная версия альтернативной галереи.
- `style.css` - единые стили сайта.
- `server.js` - HTTP-сервер, API и выдача статических файлов.
- `package.json` - точка запуска через `npm start`.
- `data/user-data.json` - файловое хранилище пользовательских данных.
- `img/` - изображения для стандартной галереи.
- `photo_for_profile/` - логотип, иконки и вспомогательные изображения.

## 3. Технологический стек

- HTML5
- CSS3
- JavaScript встраивается прямо в HTML-страницы через `<script>`
- Node.js 18+ по `package.json` и фактически совместимая более новая версия
- встроенные модули Node.js: `http`, `fs`, `path`

Отдельного сборщика, линтера, тестового раннера и frontend-framework здесь нет.

## 4. Как запустить проект

1. Установить Node.js версии 18 или выше.
2. Открыть терминал в корне проекта.
3. Выполнить `npm start`.
4. Открыть в браузере `http://localhost:3000`.

Скрипт запуска берется из `package.json`:

```json
{
  "scripts": {
    "start": "node server.js"
  }
}
```

Если открыть HTML-файлы напрямую без сервера, части сайта будут недоступны, потому что `profile.html` и `gallery.html` обращаются к `/api/...`.

## 5. Что делает сервер

`server.js` выполняет две роли:

- отдает статические файлы сайта;
- принимает и сохраняет пользовательские данные через API.

Сервер:

- работает на порту `3000`, если не задан `PORT`;
- автоматически создает папку `data/` и файл `data/user-data.json`, если их нет;
- фильтрует и нормализует входные данные перед записью;
- ограничивает размер тела запроса до 30 MB;
- защищает выдачу файлов от простого выхода за пределы корневой папки.

## 6. API проекта

### `GET /api/health`

Проверка, что сервер жив. Возвращает JSON вида:

```json
{
  "ok": true,
  "time": "ISO-date"
}
```

### `GET /api/user-data`

Возвращает весь объект пользовательских данных:

```json
{
  "profile": {
    "nickname": "",
    "email": "",
    "phone": ""
  },
  "services": [],
  "gallery": [],
  "updatedAt": null
}
```

### `PUT /api/user-data/profile`

Сохраняет профиль. Ожидает объект:

```json
{
  "nickname": "string",
  "email": "string",
  "phone": "string"
}
```

### `PUT /api/user-data/services`

Сохраняет список сервисов. Принимает либо массив, либо объект с полем `services`.

Структура элемента:

```json
{
  "id": "string",
  "url": "https://example.com/",
  "label": "example.com",
  "createdAt": "ISO-date"
}
```

### `PUT /api/user-data/gallery`

Сохраняет список пользовательских фото. Принимает либо массив, либо объект с полем `gallery`.

Структура элемента:

```json
{
  "id": "string",
  "src": "data:image/...",
  "name": "filename.png",
  "createdAt": "ISO-date"
}
```

## 7. Схема хранения данных

Все серверные пользовательские данные лежат в `data/user-data.json`.

### `profile`

Реально используется на `profile.html`.

- `nickname` - имя пользователя
- `email` - email
- `phone` - телефон

Пароль в файл не записывается.

### `services`

Реально используется на `profile.html`.

Каждый элемент содержит:

- `id` - идентификатор записи
- `url` - нормализованный URL
- `label` - подпись ссылки
- `createdAt` - дата создания

### `gallery`

Реально используется на `gallery.html`.

Каждый элемент содержит:

- `id` - идентификатор фото
- `src` - изображение в формате Data URL
- `name` - имя файла
- `createdAt` - дата добавления

### `updatedAt`

Обновляется сервером при записи профиля, сервисов или галереи.

## 8. Как данные проходят по системе

### Профиль

1. Пользователь заполняет форму на `profile.html`.
2. Браузер валидирует поля средствами HTML.
3. Встроенный скрипт отправляет `PUT /api/user-data/profile`.
4. Сервер очищает и обрезает поля через `sanitizeProfile`.
5. Результат сохраняется в `data/user-data.json`.

### Мои сервисы

1. Пользователь вводит ссылку в `profile.html`.
2. На клиенте URL дополняется протоколом `https://`, если его нет.
3. Скрипт не дает добавить дубликат в текущем списке.
4. Данные уходят в `PUT /api/user-data/services`.
5. Сервер повторно валидирует URL, отбрасывает некорректные записи и дубликаты.

### Галерея

1. Пользователь выбирает изображения в `gallery.html`.
2. Скрипт проверяет тип и размер файла.
3. Каждый файл читается через `FileReader` как Data URL.
4. Массив отправляется в `PUT /api/user-data/gallery`.
5. Сервер принимает только `src`, начинающийся с `data:image/`.
6. Данные записываются в JSON-файл.

## 9. Где хранится логика страниц

Важная особенность проекта: JavaScript не вынесен в отдельные `.js`-файлы.

- `index.html` содержит логику FAQ и мини-квиза.
- `profile.html` содержит загрузку профиля, предпросмотр и управление сервисами.
- `gallery.html` содержит загрузку, отображение и удаление пользовательских фото.
- `login.html` содержит демо-вход, переключение видимости пароля и сохранение email в `localStorage`.

Это удобно для учебного проекта, но усложняет масштабирование и повторное использование кода.

## 10. Что реально работает, а что демонстрационное

### Реально сохраняется на сервере

- профиль из `profile.html`;
- список сервисов из `profile.html`;
- пользовательские фото из `gallery.html`.

### Хранится только локально в браузере

- лучший результат квиза на `index.html` через `localStorage`;
- email из `login.html`, если включен флажок "Запомнить email".

### Демонстрационное поведение

- `login.html` не проверяет учетные данные на сервере;
- после "входа" страница просто переводит пользователя на `profile.html`;
- `registration.html` отображает форму, но не отправляет данные в backend и не создает аккаунт.

## 11. Важные ограничения

- Базы данных нет, данные лежат в одном JSON-файле.
- Многопользовательский режим не реализован.
- Пароль из профиля не сохраняется намеренно.
- Аутентификация и авторизация отсутствуют.
- Обработка данных рассчитана на учебный, а не промышленный сценарий.
- Изображения хранятся в Data URL, поэтому файл хранилища быстро растет.

## 12. Подводные камни

- `gallery.html` ограничивает пользовательскую галерею до 12 фото и до 2 MB на файл.
- На сервере `sanitizeGallery` допускает до 50 фото, но клиентский лимит ниже. Это важно помнить при ручной отправке данных.
- Раздел "Мои сервисы" принимает только `http://` и `https://`.
- Сервер дополнительно убирает дубликаты ссылок по нормализованному URL.
- В проекте нет build-step: любое изменение HTML, CSS или `server.js` сразу влияет на результат.
- `TECHNICAL_SPECIFICATION.md` может открываться в неверной кодировке в некоторых окружениях.
- В `data/user-data.json` сейчас есть реальные персональные данные и большой base64-блок изображения. Перед публикацией, отправкой или показом проекта файл нужно анонимизировать.
- Альтернативные галереи `Mod_gallery.html` и `Mod_gallery_dark.html` ссылаются на часть файлов вида `img/1.png`, `img/3.png`, `img/4.png` и другие, которых сейчас нет в папке `img/`. Это может давать битые изображения.
- В именах некоторых файлов есть опечатки и нестандартный регистр, например `Cofedensi.html` и `Contats.HTML`. При переносе на другие среды это лучше учитывать.

## 13. Что стоит проверить после изменений

- открывается ли главная страница и навигация;
- работает ли `GET /api/health`;
- сохраняется ли профиль;
- добавляются и удаляются ли сервисы;
- загружаются и удаляются ли пользовательские фото;
- не появились ли ошибки в консоли браузера;
- не сломались ли ссылки на ассеты и служебные страницы.

## 14. Краткие рекомендации по развитию

- вынести JavaScript в отдельные файлы;
- разделить серверную и клиентскую логику по папкам;
- убрать реальные персональные данные из репозитория;
- заменить JSON-хранилище на базу данных, если проект будет расти;
- добавить настоящую регистрацию и авторизацию, если они нужны не только для демонстрации.
