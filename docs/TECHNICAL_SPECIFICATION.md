# Техническое задание на разработку веб-платформы FrameFlow

## 1. Общие сведения

**Наименование системы:** FrameFlow  
**Тип системы:** учебный многостраничный веб-сайт с клиентской логикой и Node.js-сервером для хранения пользовательских данных  
**Назначение:** создание демонстрационной платформы для фотографа-любителя с возможностью просмотра информации, ведения профиля, управления ссылками на сервисы и работы с галереей изображений.

## 2. Основание для разработки

Разработка выполняется в рамках учебного проекта. Сайт создаётся как пример комплексного веб-ресурса, в котором объединены:

- статические страницы с общей навигацией;
- интерактивные формы;
- клиентская валидация данных;
- файловое серверное хранилище без подключения базы данных;
- работа с изображениями и пользовательским контентом.

## 3. Цель разработки

Целью является создание функционального и наглядного веб-сайта, который позволяет:

- представлять проект и его возможности;
- сохранять и редактировать данные профиля пользователя;
- добавлять и удалять ссылки на внешние сервисы;
- просматривать и наполнять галерею изображениями;
- предоставлять справочную информацию о проекте, контактах и политике конфиденциальности.

## 4. Область применения

Система предназначена для:

- учебной демонстрации навыков веб-разработки;
- презентации портфолио или проекта фотографа;
- показа логики работы многостраничного сайта;
- использования в локальной среде с запуском Node.js-сервера.

## 5. Состав системы

### 5.1. Клиентская часть

Клиентская часть реализуется на HTML, CSS и JavaScript и включает следующие страницы:

- `index.html` — главная страница;
- `profile.html` — личный кабинет пользователя;
- `gallery.html` — пользовательская галерея;
- `login.html` — страница входа;
- `registration.html` — страница регистрации;
- `Cofedensi.html` — политика конфиденциальности;
- `Contats.HTML` — страница контактов;
- `Mod_gallery.html` — альтернативная версия галереи;
- `Mod_gallery_dark.html` — тёмная версия галереи.

### 5.2. Серверная часть

Серверная часть реализуется на Node.js и выполняет следующие функции:

- выдача статических файлов сайта;
- обработка API-запросов;
- чтение и запись пользовательских данных в JSON-файл;
- проверка структуры принимаемых данных.

### 5.3. Хранилище данных

Данные хранятся в файле:

- `data/user-data.json`

Хранилище используется для сохранения:

- профиля пользователя;
- списка сервисов;
- галереи пользовательских изображений;
- времени последнего обновления.

## 6. Структура сайта

### 6.1. Главная страница

Главная страница должна содержать:

- логотип и верхнее меню;
- краткое описание платформы;
- блок с ключевыми возможностями;
- информационный раздел об архитектуре проекта;
- таймлайн развития;
- FAQ-блок;
- мини-квиз;
- футер со ссылками на политику и контакты.

### 6.2. Профиль пользователя

Страница профиля должна содержать:

- форму ввода имени пользователя;
- поле email;
- поле пароля;
- поле телефона;
- блок предпросмотра введённых данных;
- раздел «Мои сервисы» для добавления ссылок;
- список сохранённых сервисов с возможностью удаления.

### 6.3. Галерея

Страница галереи должна обеспечивать:

- отображение стандартной галереи;
- загрузку пользовательских фотографий;
- проверку типа файла и размера;
- сохранение изображений в серверное хранилище;
- удаление загруженных изображений;
- обновление списка без перезагрузки страницы.

### 6.4. Вход и регистрация

Страницы входа и регистрации должны обеспечивать:

- отображение формы авторизации;
- отображение формы регистрации;
- базовую клиентскую проверку заполнения полей;
- визуальное соответствие единому стилю сайта.

### 6.5. Служебные страницы

Страницы политики конфиденциальности и контактов должны содержать:

- текстовую информацию о проекте;
- описание принципов хранения данных;
- контактные ссылки и каналы связи.

## 7. Функциональные требования

### 7.1. Навигация

Сайт должен предоставлять единое меню перехода между основными разделами:

- главная;
- вход;
- профиль;
- галерея;
- контакты;
- политика конфиденциальности.

### 7.2. Работа с профилем

Система должна:

- принимать имя пользователя, email, пароль и телефон;
- проверять корректность введённых данных;
- сохранять только безопасный набор данных;
- не записывать пароль в постоянное хранилище;
- показывать статус успешного сохранения или ошибку.

### 7.3. Работа с сервисами

Система должна:

- принимать ссылку на внешний сервис;
- нормализовать URL при необходимости;
- допускать только `http` и `https`;
- исключать дубли;
- отображать список добавленных сервисов;
- позволять удалять записи.

### 7.4. Работа с галереей

Система должна:

- принимать изображения из браузера;
- ограничивать количество и размер файлов;
- отображать загруженные изображения в интерфейсе;
- удалять ненужные фото;
- сохранять изображения через серверный API.

### 7.5. FAQ и квиз

Главная страница должна содержать:

- FAQ с раскрывающимися ответами;
- мини-квиз по функционалу сайта;
- подсчёт результата;
- отображение лучшего результата локально в браузере.

### 7.6. Таймлайн проекта

Сайт должен показывать этапы развития FrameFlow в виде последовательного списка с описанием каждого этапа.

## 8. Нефункциональные требования

### 8.1. Производительность

- страницы должны открываться без задержек в локальной среде;
- интерфейс должен корректно обновляться без перезагрузки при сохранении данных;
- система должна выдерживать типовой учебный объём контента.

### 8.2. Надёжность

- сервер должен автоматически создавать файл данных при его отсутствии;
- некорректные данные должны отклоняться или приводиться к безопасному виду;
- ошибки сохранения должны отображаться пользователю.

### 8.3. Безопасность

- пароль не должен сохраняться в постоянное хранилище;
- ссылки должны проверяться на допустимый формат;
- изображения должны ограничиваться по размеру;
- сервер должен защищать файловую систему от выхода за пределы рабочей директории.

### 8.4. Совместимость

- сайт должен работать в современных браузерах;
- интерфейс должен быть адаптивным;
- верстка должна корректно отображаться на экранах разных размеров.

### 8.5. Поддерживаемость

- код должен быть разделён на HTML, CSS и JavaScript;
- логика хранения данных должна быть вынесена на сервер;
- структура сайта должна позволять дальнейшее расширение.

## 9. Сведения о данных

### 9.1. Структура пользовательских данных

Хранимый объект должен включать:

- `profile`
  - `nickname`
  - `email`
  - `phone`
- `services`
  - `id`
  - `url`
  - `label`
  - `createdAt`
- `gallery`
  - `id`
  - `src`
  - `name`
  - `createdAt`
- `updatedAt`

### 9.2. Ограничения на данные

- профиль должен содержать только текстовые поля безопасной длины;
- сервисы должны содержать только корректные URL;
- изображения должны передаваться как `data:image/...`;
- количество сервисов и фото должно ограничиваться для защиты от переполнения файла.

## 10. API системы

Система должна поддерживать следующие маршруты:

- `GET /api/health` — проверка работоспособности;
- `GET /api/user-data` — получение всех пользовательских данных;
- `PUT /api/user-data/profile` — сохранение профиля;
- `PUT /api/user-data/services` — сохранение списка сервисов;
- `PUT /api/user-data/gallery` — сохранение галереи.

## 11. Требования к интерфейсу

Интерфейс должен:

- использовать единый стиль для всех страниц;
- содержать логотип и навигацию в верхней части;
- иметь читаемую типографику;
- поддерживать визуальные статусы успеха и ошибки;
- быть понятным для пользователя без дополнительной инструкции.

## 12. Требования к развёртыванию

Для запуска системы требуется:

- установленный Node.js версии 18 или выше;
- запуск команды `npm start`;
- доступ к локальному серверу по адресу `http://localhost:3000` или по порту, заданному переменной окружения `PORT`.

## 13. Ограничения проекта

Проект не предусматривает:

- полноценную серверную аутентификацию;
- работу с базой данных;
- многопользовательский режим;
- облачное хранение файлов;
- сложную систему ролей и прав доступа.

## 14. Критерии приемки

Система считается реализованной, если:

- открываются все основные страницы сайта;
- профиль сохраняется через API;
- список сервисов добавляется и удаляется;
- изображения загружаются и сохраняются в галерею;
- FAQ и квиз работают корректно;
- данные сохраняются между перезапусками сервера;
- интерфейс не теряет читаемость на мобильных и десктопных экранах.

## 15. Заключение

Данное техническое задание определяет состав, назначение, требования и ограничения проекта FrameFlow. Структура сайта ориентирована на учебную демонстрацию полного цикла работы веб-платформы: от многостраничного интерфейса до серверного хранения пользовательских данных.
