# REPORT_COMPLEXITY.md

## Отчёт по сложности проекта FrameFlow

### 1) Краткое описание проекта
FrameFlow — это учебная многофункциональная платформа для фотографа-любителя, реализованная на HTML, CSS, JavaScript и Node.js. Архитектурно проект построен как frontend + файловый backend без базы данных. Главная цель проекта — продемонстрировать не только базовую верстку, но и содержательный интерактивный функционал, работу с пользовательскими данными, валидацию и модульный подход к развитию сайта.

Несмотря на отсутствие backend, проект включает несколько взаимосвязанных разделов: главная страница с информационными блоками, галерея с загрузкой фото, личный кабинет, раздел «Мои сервисы», контакты и политика конфиденциальности. В рамках отчётного усложнения добавлены дополнительные модули, которые повышают как информационную насыщенность, так и функциональную глубину интерфейса.

---

### 2) Уже реализованные сложности (до текущего этапа)

#### 2.1. Галерея с пользовательской загрузкой
На странице галереи реализована работа с файлами через `input type="file"` и `FileReader`.

Ключевые сложности:
- асинхронное чтение локальных файлов и преобразование их в Data URL;
- хранение изображений на Node.js сервере в `data/user-data.json` через API `PUT /api/user-data/gallery`;
- ограничение максимального количества пользовательских фото (12 шт.);
- ограничение размера файла (до 2 MB);
- фильтрация по MIME-типу (`image/*`);
- обработка ошибок (невалидный тип, превышение размера, ошибка при записи на сервер);
- динамический рендер карточек без перезагрузки страницы;
- удаление загруженных фото с мгновенным обновлением интерфейса.

#### 2.2. Личный кабинет (профиль)
В профиле реализована форма персональных данных с валидацией и локальным сохранением.

Ключевые сложности:
- HTML5-валидация полей (`required`, `type=email`, `pattern` для телефона);
- программная проверка `checkValidity/reportValidity`;
- сохранение только безопасного набора данных (`nickname`, `email`, `phone`) в `ff_profile_v1`;
- отказ от сохранения пароля в persistent-хранилище;
- восстановление данных после перезагрузки страницы;
- live-предпросмотр введённых данных.

#### 2.3. Раздел «Мои сервисы»
В личный кабинет встроен отдельный модуль ссылок на внешние фотосервисы.

Ключевые сложности:
- валидация URL с использованием `new URL(...)`;
- нормализация ссылки (автодобавление `https://`, если пользователь ввёл домен без протокола);
- контроль допустимых протоколов (только `http/https`);
- проверка на дубликаты после нормализации;
- хранение списка ссылок в `ff_user_services_v1`;
- интерактивный список: добавление, отображение, удаление;
- отдельные статусные сообщения для ошибок и успешных операций.

#### 2.4. Стилизация и адаптивность
Сайт оформлен в единой дизайн-системе на CSS-переменных с тематикой под логотип.

Ключевые сложности:
- централизованная палитра через `:root` переменные;
- единый стиль карточек, форм, кнопок, галереи и навигации;
- адаптивные правила для мобильных экранов;
- консистентность интерфейса между страницами;
- поддержка визуальной иерархии в условиях насыщенного контента.

### 3) Новые сложности, добавленные в рамках отчётного этапа

#### 3.1. Расширенный информационный блок на главной
Главная страница дополнена крупным текстовым разделом об архитектуре проекта, модели хранения, ограничениях и пользовательских сценариях.

Что усложнилось:
- структурирование большого объёма текста в читаемые модули;
- баланс между информационной плотностью и UX;
- согласованная семантическая верстка (section/article/h2/h3/p);
- сохранение единого визуального стиля для длинного контента.

#### 3.2. Таймлайн развития проекта
Добавлен отдельный модуль «Таймлайн», показывающий этапы развития функционала от базовой версии до расширенной.

Что усложнилось:
- проектирование линейной истории продукта;
- визуальное отображение этапов как самостоятельного интерфейсного компонента;
- создание reusable-стиля для карточек этапов;
- адаптация таймлайна под мобильный экран.

#### 3.3. FAQ-аккордеон
На главной внедрён FAQ-блок с раскрывающимися ответами.

Что усложнилось:
- логика открытия/закрытия элементов по клику;
- хранение состояния раскрытия в DOM-классах;
- чёткое разделение интерактивной зоны (кнопка вопроса) и контента ответа;
- организация большого числа вопросов без перегрузки страницы.

#### 3.4. Учебный мини-квиз
Добавлен тест из 7 вопросов по функционалу платформы с автоподсчётом результата.

Что усложнилось:
- построение формы с несколькими группами radio-вопросов;
- проверка полноты заполнения;
- вычисление суммы правильных ответов;
- категоризация уровня пользователя по диапазонам баллов;
- сохранение лучшего результата в `localStorage['ff_quiz_best_v1']`;
- отображение сохранённого рекорда после перезагрузки;
- сброс состояния формы и результата.

### 4) Технические ограничения файловой Node.js-реализации
Проект осознанно реализован без базы данных, с хранением в JSON-файле. Это обеспечивает простоту развёртывания, но ограничивает масштабирование и уровень безопасности.

Основные ограничения:
- серверные пользовательские данные зависят от одного файла `data/user-data.json`;
- при удалении или повреждении JSON-файла данные требуют восстановления из резервной копии;
- нет централизованной авторизации и контроля доступа;
- нет серверной валидации и журналирования действий;
- есть практический лимит размера JSON-файла и нагрузка на канал при передаче Data URL изображений;
- невозможно реализовать многопользовательский режим и синхронизацию между устройствами без backend.

### 5) Вывод
По итогам усложнения проект FrameFlow вышел за рамки «простого учебного лендинга» и стал полноценной демонстрационной платформой с несколькими независимыми модулями и общей архитектурной связностью. В проекте присутствуют:
- модульная структура интерфейса;
- работа с пользовательскими данными через Node.js API и JSON-хранилище;
- обработка файлов и валидация форм;
- интерактивные компоненты (FAQ, сервисы, квиз);
- расширенный текстовый контент для отчётной презентации;
- документированная фиксация ограничений и рисков.

С инженерной точки зрения проект демонстрирует практику построения сложного frontend-приложения с лёгким Node.js backend в условиях файлового хранения, а также даёт базу для дальнейшего роста: подключение полноценной БД, введение аутентификации и разграничения доступа.
